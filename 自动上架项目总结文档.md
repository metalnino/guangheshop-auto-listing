# 自动上架商品项目总结与维护手册

本文件是对 `自动上架商品` 项目的技术架构、菜单化工作流、数据库映射关系、生图联动机制以及后期维护要点的完整汇总，供开发和维护人员参考。

---

## 📌 一、项目概述
本项目的核心目标是**自动化商品上架流程**。它能将包含商品名称、规格、价格、库存及图片的 Excel 数据源，批量转化为数据库内的多表商品记录，并自动上传本地图片至腾讯云 COS。

### 核心技术栈
- **编程语言**：Python 3.10 ~ 3.12
- **数据库**：MySQL / PyMySQL (连接池、事务控制)
- **云服务**：腾讯云 COS (腾讯云对象存储 Python SDK)
- **表格处理**：openpyxl

## ⚙️ 三、菜单工作流与运行模式设计

系统采用控制台交互菜单方式运行，包含以下核心交互选项：

```text
============================================================
  自动商品上架系统主菜单：
============================================================
  [1] 拉取 AI 生图数据 (导入至 pending_upload)
  [2] 执行森屿家上架 (包含批量上架/单独上架)
  [q] 退出
============================================================
```

### 1. 流程一：拉取 AI 生图数据
- **原理与逻辑**：
  1. 系统会读取配置化路径（如 `D:\work\AI\AI生图流程化项目_v2\output`）中的生图成品。
  2. 分析目录中包含哪些生成批次，并向用户展示列表。
  3. 用户选择批次号（如输入 `7` 或 `1-3`）进行拉取。
- **重复检查机制（批次级一次性提示）**：
  - 系统同时扫描本地 `pending_upload` 和 `completed_upload` 目录。
  - 若检测到目标批次文件夹前缀（形如 `{b_id}-`）或 `batch_info.json` 内的 `batch_id` 匹配，会汇总列出已存在的目录，并**仅发出一次询问提示**：
    `是否仍要重复执行拉取该批次？(y/n)`
  - 如果用户输入 `n`，则终止当前批次导入并退回批次选择界面；若按回车（默认）或输入 `y` 则强制执行，避免单图重复判定打断体验。
- **导入与表格初始化**：
  - 将对应的生图成品（包括 scenes 目录下的图片）拷贝到暂存区，目标文件夹以批次号为前缀命名：`pending_upload/{batch_id}-{plant_name}_{timestamp}`。
  - 生成对应批次的 `batch_info.json` 写入图片目录。
  - 自动向 `pending_upload/商品上架数据.xlsx` 中追加对应的待上架记录，其中：
    - `商品名称`、`规格值` 和 `图片目录` 字段根据 AIGC 数据包自动解析填充。
    - `单位` 默认填 `盆`，`公共/私有` 默认填 `公共`。
    - **`团长名称`（森屿家用户）留空不预填**，交由用户在 Excel 中手动选取或输入。

### 2. 流程二：执行森屿家上架（老逻辑）
- **数据源识别**：
  - 如果存在暂存数据，优先读取 `pending_upload/商品上架数据.xlsx` 进行上架，并进行“主图/轮播图”的关联和自动归档。
  - 如果暂存数据为空，系统会进行**空数据拦截**，给出清晰提示，避免闪退或直接关闭窗口，用户可按回车返回主菜单。
  - 若无暂存数据，则转为手动模式读取根目录下的 `商品上架模板.xlsx`。
- **COS 自动上传**：
  - 自动扫描解析图片目录，上传至腾讯云 COS 存储桶中，无需手动处理图片上传。
- **完成/失败归档控制**：
  - 上架成功的记录行会从 Excel 表格中自动删除，且其对应的图片文件夹移动至 `completed_upload/` 下归档。
  - 上架失败的记录保留，并在最右侧写入报错原因。

## 🗄️ 四、数据库表结构及写入逻辑

每次成功上架一个商品，系统会在一个**数据库事务**中依次向以下 6 张表写入/更新数据，如果中途失败，事务会自动回滚，确保数据库脏数据清空：

1. **`litemall_goods` (商品主表)**：
   - 插入商品基本信息，生成自增的 `goods_id`，并使用雪花 ID / 时间戳算法生成唯一商家编码 `goods_sn`。
2. **`litemall_goods_specification` (商品规格属性值表)**：
   - 创建规格属性（如“规格=11cm”），获取自增的 `spec_id`。
3. **`litemall_goods_product` (商品SKU/库存单元表)**：
   - 将主商品与规格值关联，生成唯一的 `product_id` (即 SKU ID)，写入对应的售价、批发起步价。
4. **`litemall_goods_attribute` (商品参数表)**：
   - 写入商品产地、规格等静态展示参数。
   - **团长名称对应数据**：通过 Excel 中的“团长名称”字段，自动从数据库中查询解析出对应的 `creator_id` 写入。
5. **`litemall_goods_region` (商品投放城市关联表)**：
   - 将商品的 `goods_id` 与对应城市的区域代码（如 `320500` 苏州市）进行绑定。支持多城市绑定。
6. **`litemall_goods_inbound_log` (商品库存入库日志表)**：
   - 记录初次入库日志，将初始库存计入入库流转，触发库存计算逻辑。

## 🛠️ 五、核心维护与防错机制

为了保证系统的健壮性，项目中包含以下专门的防错机制设计，在后期维护修改时需注意保留：

### 1. 动态下拉数据源关联
- **团长名称/森屿家用户（F 列）下拉源数据**：
  - 系统内部内置了 `get_all_creators()` 逻辑，启动时会从 `z100h_wx.wx_user` 联表 `guangheshop.litemall_user` 实时提取所有已绑定的森屿家成员名称。
  - 同时在代码中强制对“吴鑫霞”和“李琳”进行兜底确保安全。
  - 将去重、排序后的用户列表自动录入隐藏工作表 `下拉源数据` 的 C 列，使用 `DataValidation` 为 Excel `F3:F200` 单元格绑定下拉菜单。
  - 表格第二行注释更改为：`森屿家用户，如李琳`。
- **省市级快捷联选下拉（J, K~O 列）**：
  - 根据 `generate_template.py` 中定义的地理省份数据包，在 `下拉源数据` 的 A、B 列中动态绑定快捷省份与城市下拉菜单。

### 2. 交互式 Excel 占用锁定拦截
- **机制**：
  - 在正式写入数据库前，脚本会尝试以独占写模式打开 Excel。
  - 如果抛出 `IOError`，说明文件正在被占用，程序会**暂停运行**并给出提示：
    `⚠️ 检测到 Excel 文件被锁定或被占用，请先关闭 Excel/WPS 软件！`
  - 用户关闭软件后在终端中**按下回车 [Enter] 即可重试检测**，或者输入 `q` 安全退出。

### 3. Windows 命令行（CMD）字符编码防乱码
- **机制**：
  - 启动器 `run_auto_listing.bat` 及 `install.bat` 内部被限制为**纯 ASCII 字符（英文）**，杜绝了解析冲突。
  - 脚本开头显式设置了 `PYTHONIOENCODING=utf-8` 和 `PYTHONUTF8=1`，确保 Python 往命令行输出的日志信息（包括 emoji 如 🌱、✓、★）都能被正常解析输出。

### 4. 日志自动归档
- **机制**：
  - 启动时自动建立 `logs/` 文件夹。
  - 脚本初始化时会通过 glob 匹配将根目录下历史残留的 `auto_listing_*.log` 文件搬运至 `logs/` 目录下，并以 `logs/auto_listing_YYYYMMDD_HHMMSS.log` 生成当前运行日志。

## 🚨 六、维护开发常见问题与注意点

1. **外部配置化管理**：
   - 数据库连接信息、腾讯云 COS 密钥、外部生图拉取目录路径（`AI_OUTPUT_DIR`）以及商品默认基础字段均定义在 **`config.py`** 配置文件中。
   - 如果更换运行环境或更换生产库，请直接编辑 `config.py` 相应的字典变量，无需修改主体程序源码。
2. **依赖更新**：
   - 增加新功能如果引入了新的 pip 包，必须同步写入 `requirements.txt` 中，以保证 `install.bat` 能正常工作。
